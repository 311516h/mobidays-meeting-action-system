# Mobidays AI Tech Lab Proposal

## 1. 문제 재정의

모비데이즈에서는 광고주별 캠페인 회의 이후 담당자가 회의록을 정리하고, 회의 중 결정된 액션아이템을 별도의 트래킹 시스템으로 수기 등록하고 있다.

이 과정에서 다음과 같은 문제가 발생한다.

- 회의 종료 후 회의록 작성 및 액션아이템 정리에 30~60분의 추가 시간이 소요된다.
- 한국어 회의 특성상 책임자(R&R)가 암묵적으로 표현되어 액션아이템 누락이 발생한다.
- 회의 내용은 남지만 후속 업무 추적이 가능한 구조화 데이터는 생성되지 않는다.

본 프로젝트는 단순 회의록 생성보다 액션아이템 누락 방지에 우선순위를 둔다. 회의록은 사람이 직접 검토할 수 있지만, 액션아이템이 누락되면 업무 지연, 책임 불명확, 추가 커뮤니케이션 비용이 발생한다.

따라서 본 시스템은 회의 내용을 구조화된 업무 데이터(Action Item)로 변환하여 후속 업무 관리가 가능하도록 하는 것을 핵심 목표로 한다.

## 2. 시스템 아키텍처

### End-to-End Flow

Transcript JSON -> Preprocessing -> Chunking -> LLM Action Item Extraction -> Schema Validation -> DuckDB Storage -> Slack Payload Generation -> Dashboard Analytics

```text
회의 transcript
  -> 화자/용어/중복 발화 정규화
  -> 의미 단위 chunk 생성
  -> LLM 기반 액션아이템 추출
  -> Pydantic 스키마 검증 및 재시도
  -> DuckDB 적재
  -> Slack payload 생성
  -> Streamlit 분석 대시보드
```

### 기술 선택 근거

#### DuckDB

- 설치가 간단하고 별도 서버가 필요 없다.
- 분석성 쿼리에 최적화되어 대시보드 구성에 적합하다.
- PoC 규모에서는 Postgres 대비 운영 복잡도가 낮다.

#### Streamlit

- 빠르게 데이터 기반 대시보드를 구현할 수 있다.
- 데이터 파이프라인 결과를 시각적으로 검증하기 쉽다.

#### Pydantic

- LLM 출력 스키마를 강제할 수 있다.
- 잘못된 JSON 응답을 조기에 탐지할 수 있다.
- 신뢰 가능한 데이터 자산 구축에 유리하다.

#### Transcript 기반 처리

- STT 품질 변수를 줄이고 액션아이템 추출 설계에 집중할 수 있다.
- 제공된 검증용 transcript를 기준으로 멱등적인 파이프라인 검증이 가능하다.

## 3. 데이터 스키마 설계 근거

본 PoC는 회의 원문과 LLM 추출 결과를 분리 저장하는 구조를 선택했다. 회의 발화는 감사와 재처리를 위해 원본성을 유지하고, 액션아이템은 후속 업무 관리에 바로 사용할 수 있는 구조화 데이터로 관리한다.

### 주요 테이블

- `meetings`: 회의 단위 메타데이터를 저장한다. 광고주, 캠페인, 회의일, 입력 타입을 관리한다.
- `utterances`: transcript의 발화 단위를 저장한다. 화자, 역할, 발화 시간, 원문, 정규화 텍스트를 분리해 추적성을 확보한다.
- `chunks`: LLM 입력 단위를 저장한다. 발화 전체를 한 번에 넣지 않고 의미 단위로 나누어 문맥 손실과 토큰 낭비를 줄인다.
- `action_items`: 최종 업무 데이터를 저장한다. 담당자, 업무, 기한, 우선순위, 상태, confidence, 근거 발화를 함께 저장한다.

정규화는 원본 회의 데이터의 추적성과 재처리 가능성을 위해 적용했다. 다만 대시보드에서 바로 조회할 수 있도록 `action_items`에는 owner, task, due_date, confidence 같은 분석 필드를 명시적으로 둔다. 이는 PoC 환경에서 복잡한 join 없이 의사결정 지표를 빠르게 확인하기 위한 선택이다.

## 4. Before / After 임팩트 추정

현재 회의 후 회의록 정리와 액션아이템 수기 등록에는 회의 1건당 약 30~60분이 소요된다. 본 시스템 도입 후에는 transcript 적재, 액션아이템 초안 생성, 담당자 검토 흐름으로 바뀌므로 담당자의 수작업 시간을 10~15분 수준으로 줄이는 것을 목표로 한다.

100명 규모 조직에서 주당 30건의 캠페인 회의가 발생한다고 가정하면, 기존에는 주당 약 15~30시간의 정리 시간이 필요하다. 자동 추출 후 검토 방식으로 전환하면 주당 약 5~7.5시간으로 줄어들어, 주당 10~22.5시간의 운영 시간을 절감할 수 있다.

품질 측면에서는 액션아이템마다 source utterance와 confidence를 함께 제공해 낮은 신뢰도 항목만 사람이 우선 검토할 수 있도록 한다. 이를 통해 모든 회의록을 다시 읽는 방식보다 누락 검토 범위를 줄이고, 암묵적 R&R 표현을 구조화해 액션아이템 누락률을 낮추는 것을 기대한다.

## 5. 실패 시나리오와 대응

### 시나리오 1: LLM이 존재하지 않는 액션아이템을 생성하는 경우

LLM 출력에는 반드시 `source_utterance`와 `confidence`를 포함한다. 원문 근거가 비어 있거나 스키마를 만족하지 못하는 결과는 Pydantic 검증 단계에서 제외하거나 재시도한다. confidence가 낮은 항목은 대시보드에서 드릴다운 대상으로 표시해 사람이 검토한다.

### 시나리오 2: 한국어 회의의 암묵적 담당자 표현을 잘못 매핑하는 경우

"제가 챙길게요", "그건 콘텐츠 쪽에서 보면 될 것 같아요"처럼 담당자가 명시되지 않은 표현은 speaker와 role 정보를 함께 사용해 owner 후보를 추론한다. 추론 근거는 reasoning 필드에 남기고, 확신이 낮은 경우 owner를 확정하지 않고 검토 대상으로 둔다.

### 시나리오 3: 파이프라인을 여러 번 실행해 데이터가 중복 적재되는 경우

회의, 발화, chunk, action item에 deterministic id를 부여한다. 같은 transcript를 다시 처리해도 동일한 primary key로 upsert되도록 설계해 중복 적재를 방지한다.
